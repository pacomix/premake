Project Scripts

  • Project scripts contains the required information needed by Premake in order to generate a Visual Studio 2008/2010 project .vc(x)proj file.
  • A project script file is basically a container of Lua global variables that are used by the main Premake script to define a project object.
  • You can create as many additional helper variables and functions in order to assign final values to the required script variables. Remember, you are programming in Lua.
  • The variables that should contain more than one value are table variables. In the project script both types exist and will be explicitly written when a table is required with the {} characters.
  • When using quotes and/or backwardslashes it is preferably to use the single quotes ' ' to define the entry because there is no need to escape the quotes and backward slashes characters.
  • Remember that when using environment variables it is preferably to always use quotes to prevent wrong directories be specified when containing spaces.
  • Head to every section in order to get a detailed description of the available variables.
  • After reading this chapter you can go to trunk\vBase\vBase and open the associated Prj-vBase.lua file to see how a project script should look like.
  • The minimum required information for a project script is:
    • Output project file location.
    • Project name.
    • Project GUID.
    • Output target directory.
    • Output target name.
    • Build configurations/target platforms.
    • Project's programming language.
  • EVERY project script file should be able to properly manage the generation of projects for Visual Studio 2008/2010 and XCode 3.2. In the case the project shouldn't be generated for the target IDE it must return with an error message. IE: 
 if buildSystem == "vs2008" or buildSystem == "vs2010" then
   error("Project shouldn't be generated for this IDE: " .. buildSystem)
 elseif buildSystem == "xcode3" or buildSystem == "xcode4" then
   . Project information
   . Project information
   . Project information
   . Project information
   . Project information
 else
   error("Project shouldn't be generated for this IDE: " .. buildSystem)
 end

Project's Description

  • The next paragraphs describes the sections you can define inside a project Prj-ProjectName.lua script file and contains all the information you need to generate a successful Visual Studio Project file.
  • In opposition to solution's script files here it doesn't matter the order in which the Lua variables are used.
  • Variables that have empty values can be suppressed from the script.
  • Note:
    • Though the order of the sections is not relevant at the time of generating the target IDE project file it must respect the order of the already implemented project script files so everything looks the same and be easy to maintain.
    • Though the empty variables can be deleted it is a better idea to leave them there so in the need of adding more paramaters to a project it could be easy to modify.


Minimum Required Information

  • The next content should be ALWAYS at the top and bottom of EVERY script project file: 
 -- BOF ---
 globalScriptDir = os.getcwd()
   Script Variables
   Script Variables  
   Script Variables
   Script Variables
 ConfigureBuild("Project")
 -- EOF --
  • Here you can see the minimum required information to generate a project file: 
 projectLocation           - Output directory of the generated project file.
 projectName               - Project name and name of the generated project file.
 projectGUID               - Unique project identifier. For XCode 3.2 it is ignored.
 outputFolder              - Output folder of the project. When this is specified the correct binXXXXX output folder is appended based in the build configuration name otherwise the main script will use the variables named outputFolderXXXXXXXXX.
 outputTarget              - Output target name of the project. When this is specified the correct target/import library suffix is appended to the target name based in the build configuration name otherwise the main script will use the unchanged variables named outputTargetXXXXXXXXX.
 projectBuildConfigs{}     - Project's build configurations in the format: ConfigName|TargetPlatform.
                               - This variable is later used against includeBuildConfigs to mark which build configurations should be marked as buildable.
                               - NOTE: TargetPlatform should be written in the Premake format. This is: x32, x64, Xbox360, iOS
 includeBuildConfigs{}     - Build configurations that should be marked as buildable. NOTE: Because we can also define this on a per solution basis you should only define this variable only if only it wasn't defined previously.
 projectLanguage           - Programming language of the project. C, C++, C#, Default
 buildConfigsForProject{}  - Build configurations of this project. IE: { "Debug", "Debug DX11", "Release", "Release DX11" }
 buildPlatformsForProject{}- Target platforms of this project in Premake format. IE: { "x32", "x64", "Xbox360" }
  • Some other optional parameters of interest: 
 projectNiceName           - Optional. Project name in Visual Studio. This name is the one that you will see in Visual Studio.
 projectIcon               - For C# projects. Path to the generated application's icon. This file MUST be added to the source files.
  • Next you can see a typical minimum requirements project script: 
 globalScriptDir = os.getcwd()

 tempSlnPrefix = "PRE"                             -- Helper variable to not overwrite the existent project. Only used to be able see the differences locally.
 projectLocation = "."
 projectBaseName = tempSlnPrefix .. "SPUSyncEvent" -- Helper variable to build the project name.
 projectName = projectBaseName .. buildSystemSuffix
 projectGUID = "D47436BF-79F9-41CA-8B12-E8D57C93D67E"
 outputFolder = "../../../lib/spu/"
 outputTarget = "SPUSyncEvent"
 projectLanguage = "Default"
 projectBuildConfigs = { "Debug PS3|x32", "Release PS3|x32" }
 buildConfigsForProject = { "Debug PS3", "Release PS3" }
 buildPlatformsForProject = { "x32" }

 -- We only want to include the whole project build configurations if we didn't define it previously.
 if includeBuildConfigs == nil or includeBuildConfigs == "" then
   includeBuildConfigs = projectBuildConfigs
 end

 ConfigureBuild("Project")

Project References

  • Project references are specified through:
    • projectReferences{}
  • It must use the name of a project that is inside the solution. IE: Extract from Prj-vBaseTests.lua 
 if buildSystem == "vs2008" then
   projectReferences = { "SPUDummyTask", "SPUDummyTaskDMA", "SPUStreamProcessPointerTest", "SPUStreamProcessSimpleTest", "vBase" .. buildSystemSuffix  }
 elseif buildSystem == "vs2010" then
   projectReferences = { "vBase" .. buildSystemSuffix, "vBaseTestUnitDlg" .. buildSystemSuffix }
 end
  • Related flags: #NoLinkLibDependencies, #VSUseReferencesInProjects

EnableSSE, EnableSSE2

  • Use the SSE instruction sets for floating point math.

MinWarnings, ExtraWarnings

  • Sets the warning output level.
  • MinWarnings will set the minimum warning output and ExtraWarnings will set it to the highest level.
  • If this flag is not defined the warning level by default will be Level-3.
  • Related flags: #FatalWarnings

FatalWarnings

  • Threat warnings as errors.

VSEnableFunctionLevelLinking

  • Sets the Visual Studio option Enable Function-Level Linking.

VSBufferSecurityCheck

  • Sets the Visual Studio option Buffer Security Check.

VSNoBasicRuntimeCheck

  • Sets the Visual Studio option Basic Runtime Checks to Default.
  • Note: The Basic Runtime Check parameter is set only if only Optimizations are disabled and the project is a non Managed project.

VSNoStringPooling

  • Sets the Visual Studio option Enable String Pooling to No.

VSEnableIntrinsicFunctions

  • Sets the Visual Studio option Enable Intrinsic Functions.

VSFavorSize

  • Sets Visual Studio Favor Size or Speed option to Favor Small Code.

VSFavorSpeed

  • Sets Visual Studio Favor Size or Speed option to Favor Fast Code.

VSNoOptimizeReferences

  • Sets Visual Studio References optimization to Default. Otherwise is set to Eliminate unreferenced data.

VSNoCOMDATFolding

  • Sets Visual Studio COMDAT Folding option to Default. Otherwise is set to Remove redundant COMDATs.

VSNoTargetMachine

  • Sets the Target Machine to Not set.

VSNoSubSystem

  • Sets the SubSystem to Not set.

VSUseReferencesInProjects

  • Force using the referenced project inside the project information and not as a dependency.
  • Note: This flag is automatically added when defining the references a project has. So there is no need to add this flag.

VSEnableStaticCodeAnalysis

  • Enables the static code analysis of Visual Studio 2010 Ultimate. Also see: Static Code Analysis with Visual Studio 2010

NoLinkLibDependencies

  • When this flag is set the project dependencies are not automatically linked to the project when linking.
  • If so you have to manually set them in the #Link Libraries variables.

GenerateStrippedSymbols

  • It writes another Symbols database file without private symbols.
  • It is only effective when defining the Symbols flag.

FloatFast

  • Sets the Visual Studio Floating Point Model parameter to Fast.
  • Note: If not defining either FloatFast or FloatStrict the parameter will contain the default value.

FloatStrict

  • Sets the Visual Studio Floating Point Model parameter to Strict.
  • Note: If not defining either FloatFast or FloatStrict the parameter will contain the default value.

Managed

  • Sets the project to be a Managed project using the Common Language Runtime support.
  • When defining the this flag the Minimal Rebuild and Basic Runtime Checks parameters are disabled.
  • Related flags: #VSNoBasicRuntimeCheck

MFC

  • Uses MFC libraries instead of Standard Windows Libraries.

NativeWChar, NoNativeWChar

  • Enable or disable support for the wchar data type. If no flag is specified, the toolset default will be used.

No64BitChecks

  • Disable 64-bit portability warnings. Only affecting Visual Studio versions < 2008.

NoEditAndContinue

  • Disable support for Visual Studio's Edit-and-Continue feature.

NoExceptions

  • Disable C++ exception support.
  • When not defining this flag and also not the SEH flag the exception support is set to Yes.

SEH

  • Enable structured exception handling (SEH exceptions).
  • When not defining this flag and also not the NoExceptions flag the exception support is set to Yes.

NoFramePointer

  • Disable the generation of stack frame pointers.

NoImportLib

  • Prevent the generation of an import library for a Windows DLL.

NoManifest

  • Prevent the generation of a manifest for Windows executables and shared libraries.

NoMinimalRebuild

  • Disable Visual Studio's minimal rebuild feature.
  • Minimal rebuild is only activated for debug builds that are non Managed projects.
  • Related flags: #Managed

NoIncrementalLink

  • Disable support for Visual Studio's incremental linking feature.

NoPCH

  • Disable precompiled header support. If not specified, the toolset default behavior will be used.

NoRTTI

  • Disable C++ runtime type information.

Optimize

  • Perform a balanced set of optimizations.
  • When defining this flag the Debug information level is never set to Edit & Continue.

OptimizeSize

  • Optimize for the smallest file size.
  • When defining this flag the Debug information level is never set to Edit & Continue.

OptimizeSpeed

  • Optimize for the best performance.
  • When defining this flag the Debug information level is never set to Edit & Continue.

StaticRuntime

  • Perform a static link against the standard runtime libraries.
  • Based on the Debug or Release configuration the correct link library is selected.
  • Also when having the flag MFC defined the correct options are selected.

Symbols

  • Generate debugging information.
  • The symbols file will be generated in the same place as the output target.

Unicode

  • Use Unicode character set. Otherwise Multi-Byte is used.

Unsafe

  • Enable the use of unsafe code in .NET applications.

WinMain

  • Use WinMain() as the program entry point for Windows applications, rather than the default main().

Custom Build Command

  • The Custom Build Command for specific files can be set in the next way. NOTE: Only valid for Visual Studio: 
 fileCustomBuildTool{"file(s)"}{"parameter to be set"}   -- Nested table that configures the custom build step.
                                                         --
                                                         --   "file(s)" can be:
                                                         --     Any file declared in the source files table.
                                                         --     A set of files declared with wildcards. IE: "**.shader"
                                                         --
                                                         --   "parameter to be set" can be one of:
                                                         --     CommandLine{}   - Specifies the command line
                                                         --     Outputs{}       - Specifies the output files generated by the CommandLine
                                                         --     Dependencies{}  - Specifies the file dependencies
  • Actually you have to previously declare the nested table with the amount of entries for files it will contain in the next way: 
 fileCustomBuildTool = {
   ["file_entry_000"] = { },
   ["file_entry_001"] = { },
   ["file_entry_002"] = { },
   ["file_entry_003"] = { },
      .
      .
      .
   ["file_entry_xxx"] = { }
 }
  • And then use it in the next way: 
 fileCustomBuildTool["file_entry_xxx"]["parameter to be set"] = {
   table_value_000,
   table_value_001,
      .
      .
      .
   table_value_xxx
 }
  • Example from VisionEnginePlugin: 
 local outputFilename = '$(ProjectDir)..\\..\\src\\vScript\\wrappers\\VisionLuaModule_wrapper.cpp'
 local VisionLuaModule = "../../include/vScript/wrappers/interfaces/VisionLuaModule.i"
 local depsCustomBuildTool = {
   "../../include/vScript/wrappers/interfaces/vision_types.i",
   "../../include/vScript/wrappers/interfaces/VisionLuaModule.i",
   "../../include/vScript/wrappers/interfaces/VBitmask.i",
   "../../include/vScript/wrappers/interfaces/VColor.i",
     .
     .
     .
 }

 fileCustomBuildTool = { [VisionLuaModule] = { } }

 fileCustomBuildTool[VisionLuaModule]["Outputs"     ] = { outputFilename }
 fileCustomBuildTool[VisionLuaModule]["Dependencies"] = { depsCustomBuildTool }
 fileCustomBuildTool[VisionLuaModule]["CommandLine" ] = {
   '$(ProjectDir)..\\..\\..\\external\\swigwin-2.0.3\\swig -c++ -lua -verbose -o "' .. outputFilename .. '" "$(InputPath)"',
   '$(ProjectDir)..\\..\\tools\\iswig --includestdafx.h "' .. outputFilename .. '"'
 }

Precompiled Headers

  • Precompiled header configuration can be set through the next variables. 
 sourceHeaderPCH   -- Configures the project to use this file as the Precompiled Header
 sourcePCH         -- Configures the file to be the one that generates the Precompiled Header
  • There is a little issue when using the Xbox360 as target platform in conjunction with property sheets. Not every property of the property sheet is inherited correctly and this is on of those. For this case we explicitely need to specify the Precompiled Header to use through: 
 sourceHeaderPCHXbox360  -- Sets the correct precompiled header option for Xbox360 when using property sheets.
  • Additionally you can specify which files should not make use of the precompiled header file by defining a file list in: 
 sourcesNotUsingPCH{}
  • Related flags: #NoPHC.


Project's Source Files

  • The list of files included in the project is defined in: 
 sourceFiles{}
  • To easily exclude files temporarily from a solution you can use the comment style of lua. IE: -- 
 sourceFiles = {
   "./src/navy_moves.cpp",
 --"./src/army_moves.cpp",
   "./src/phantomas.cpp",
   "./src/game_over.cpp",
 }
  • Rules:
    • The file list should be specified relative to the location of the project script file.
    • It should ABSOLUTELY respect the filename casing.
    • Although the use of wildcards are allowed in the entries WE WILL NOT USE IT because it causes some issues when excluding files from build configurations.
  • Additionaly you can exclude files from build configurations by defining them inside this variables:
  • NOTE: The variable that excludes the sources for Wii writes the entries to a output file named ignoreWii.txt at the same directory level of the project. If this file is new keep in mind that it should be added to the repository as well. 
 excludedSourcesWindows{}  excludedSourcesWindowsNoXI{}
 excludedSourcesXbox360{}
 excludedSourcesPS3{}
 excludedSourcesPSP2{}
 excludedSourcesMac{}
 excludedSourcesWii{}
  • You can add resource files to a project by making use of the next variable. When putting list of files inside this variable the files will be embedded in the project.
  • NOTE: This only applies to XCode 3.2. and Visual Studio C# projects. For Visual Studio C++ projects you can add them in the same list as the source files. 
 resourceFiles{}

 buildTempDir

 callConvention  -- Allowed values: "cdecl", "fastcall", "stdcall"
  • Project's property sheet is supported only at the moment in Visual Studio 2008.

 projectPropertySheet

 delayDllCommon    -- This variable is only intended to be use as a helper. It is not used at all when generating the project file.

  • You can specify the culture for the resource compiler through: 
 culture

Misc Variables

  • There a couple of global variable defined that are quite useful when writting project script files.
    • buildSystem - Contains the IDE for which the project is being generated. It can contain vs2008, vs2010 or xcode3.
    • buildSystemSuffix - Contains the IDE suffix that is used to name project and solutions. It can contains: 90 when generating for Visual Studio 2008 or 100 when generating for Visual Studio 2010
    • bIsEnginePlugin - This variable MUST be set to true when creating a Engine Plugin script project.
    • bIsSPU - This variable MUST be set to true when creating a PS3 SPU script project.

Wii Specifics

  • Because the Wii follows a completely different build configuration special variables had to be created for this platform.
  • The Wii configuration is composed of a Build command line, a Clean command line and preprocessor definitions. 
 buildCommandLineDebugWii{}    -- Wii build command line for Debug builds.
 buildCommandLineReleaseWii{}  -- Wii build command line for Release builds.

 cleanCommandLineDebugWii{}    -- Wii clean command line for Debug builds.
 cleanCommandLineReleaseWii{}  -- Wii clean command line for Release builds.

 definesDebugWii{}   -- Defines for Debug builds.
 definesReleaseWii{} -- Defines for Release builds.
