Solution Scripts

  • Solution scripts contains the required information needed by Premake in order to generate a Visual Studio 2008/2010 solution .sln file.
  • The required information is:
    • Location of the main generator script.
    • Solution name.
    • Script location.
    • Solution target directory.
    • Build configurations.
    • Target platforms.
    • Referenced projects and per project build configs (optional).

Sections

  • The next paragraphs describes the diferent sections of a solution Sln-SolutionName.lua script file and contains all the information you need to generate a successful Visual Studio Solution file.
  • Note: The order of the sections is crucial and should never be different from what is here explained.

Header

  • The next content should be ALWAYS at the beginning of EVERY script solution file and SHOULD BE NEVER CHANGED.
  • What is does is to locate the main script (the real one) the makes use of the variables defined for every project in each Prj-ProjectName.lua script file. 
 --------------------------------------------------------------------------------------
 globalScriptDir = nil -- It should contain the dir where the script file is located. Used for including the standard script files to generate the projects and solutions.
 
 function ConfigureBuild(element)
   if element ~= "Solution" and element ~= "Project" then
     error("\n\n\tOnly \"Solution\" or \"Project\" are allowed\n\n")
   end
 
   -- Look for the scripts location
   globalScriptDir = os.getcwd()
   local iSecurityCounter = 0
   while not os.isdir("./vABC") do
     os.chdir("..")
     iSecurityCounter = iSecurityCounter + 1
     if iSecurityCounter >= 16 then
       break
     end
   end
 
   if iSecurityCounter >= 16 then
     error("\n\n\tERROR!!!!! Global Solution Vars file can not be found.\n\tStarted from:\n\n\t\t" .. globalScriptDir .. "\n\n\tEnded in:\n\n\t\t" .. os.getcwd())
   end
   
   os.chdir("vABC/Premake")
   if element == "Solution" then
     dofile "Globals.lua"  -- Global functions and variables
     CleanVariables()
   elseif element == "Project" then
     dofile "PrjGenerator.lua" -- Generates the solution and projects.
   end
   
   os.chdir(globalScriptDir) -- Restore the previous path.
   
 end
 
 ConfigureBuild("Solution")
 --------------------------------------------------------------------------------------

Build Configurations and Target Platforms

  • The next section tells which build configurations and platforms will be present in the solution.
  • Remember: This list is exclusively for the solution and it has nothing to do with the project's build configuration list.
  • Target platforms can be:
    • x32 - 32-bit environment (Win32)
    • x64 - 64-bit environment (x64)
    • Xbox360 - Xenon platform
    • iOS - Apple iPad 
 if buildSystem == "vs2008" then -- Build configurations for Visual Studio 2008
 
   -- Define here the diferent build configuration separated by entries. IE:
   buildConfigs = { "Debug", "Debug NoXI", "Release PS3", "Release PSP2" }
   
   Define here the diferent platforms separated by entries. IE:
   buildPlatforms = { "x32", "x64", "Xbox360" }
 
 elseif buildSystem == "vs2010" then -- Build configurations for Visual Studio 210  
 
   -- Define here the diferent build configuration separated by entries. IE:
   buildConfigs = { "Debug", "Debug NoXI" }
   
   -- Define here the diferent platforms separated by entries. IE:
   buildPlatforms = { "x32", "x64" }
 
 elseif buildSystem == "xcode3" then -- Build configurations for XCode 3.2
 
   -- Define here the diferent build configuration separated by entries. IE:
   buildConfigs = { "Debug", "Debug NoXI", "etc", "etc" }
   
   -- Define here the diferent platforms separated by entries. IE:
   buildPlatforms = { "Universal32", "iOS" }
   
 end

The Solution Object

  • The next section that should appear is the Premake's solution object. This is where we tell Premake what we want for our solution. 
 solution ( "Name of the solution" )
   basedir "." -- Current directory of the Sln-SolutionName.lua script.
   location "."  -- Where to output the generated SolutionName.sln file.
   configurations { buildConfigs } --Solution's build configurations.
   platforms      { buildPlatforms } --Solution's target platforms.

Referenced Projects

  • Once we have the object solution defined we have to tell which projects will be referenced inside the solution. 
 -- Define here each project that is part of the solution. IE:
 if buildSystem == "vs2008" then 
 
   dofile "vBase/Project-vBase.lua"
   dofile "../shared/projects/SPU/SPUDummyTask.lua"
   dofile "../shared/projects/SPU/SPUDummyTaskDMA.lua"
   
 elseif buildSystem == "vs2010" then
 
   dofile "vBase/Project-vBase.lua"
   
 elseif buildSystem == "xcode3" then
 
   dofile "vBase/Project-vBase.lua"
   dofile "../shared/projects/SPU/SPUDummyTask.lua"
   dofile "../shared/projects/SPU/SPUDummyTaskDMA.lua"
   
 end

Active Build Configurations

  • The referenced projects can of course have different build configurations than the ones specified in the solution.
  • Those build configurations that doesn't match the ones defined in the solution file are automatically mapped to the first valid build configuration in the project and set to not build.
  • There is an additional mechanism that lets you skip any specific build configuration of a given project. By defining which build configurations of the project's build configuration should be build.
  • For this there are two possible options. To define which project's build configuration should be included or which ones should be excluded.
  • This is carried out by defining either the variable includeBuildConfigs or excludeBuildConfigs and it should be defined before referencing the respective project.
  • For ease of use it is preferable to use includeBuildConfigs because it gives instantly which project's build configurations will be built.
  • Note: If this is not defined all the project's build configuration will be taken in account and marked to be built. 
 if buildSystem == "vs2008" then 
 
   includeBuildConfigs = {
     "Debug|x32",    "Debug DX11|x32", "Debug PS3|x32",    "Debug PSP2|x32",   "Debug Wii|x32",
     "Release|x32",  "Release DX11|x32", "Release PS3|x32",  "Release PSP2|x32", "Release Wii|x32",
     "Debug|x64",    "Debug DX11|x64", "Release|x64",  "Release DX11|x64", "Debug|Xbox360", "Release|Xbox360" }
     
     dofile "vBaseTests/Prj-vBaseTests.lua"
     
 elseif buildSystem == "vs2010" then
 
   includeBuildConfigs = { "Release|x32" }
     
     dofile "vBaseTests/Prj-vBaseTests.lua"
     
 elseif buildSystem == "xcode3" then
 
   includeBuildCfongis = { "Debug|x32", "Release|x32" }
   
     dofile "vBaseTests/Prj-vBaseTests.lua"
 
 end

Mapping Configurations

  • There is also the possibility to map solution's build configurations to specific project's build configurations.
  • This allows you to build a different project's build configuration than the solution's build configuration when triggering a solution's build.
  • This is most needed in solutions that contains C# projects with Any CPU and not Win32/x64 configurations.
  • This is carried out by defining the variable mapBuildConfigs and it should be defined before referencing the respective project.
  • Note: The content of the variable must match the build configuration name of Visual Studio. 
 mapBuildConfigs = {
   "Debug Wii|Win32=Debug|Any CPU", 
   "Release Wii|Win32=Release|Any CPU"
 }
 dofile "RunWiiTests/Prj-RunWiiTests.lua"

Inter-project dependencies

  • Inter-project dependencies are specified through:
    • projectDependencies{}
  • It must use the name of a project that is inside the solution and like in the mapping build configurations it should be declared just before referencing the project. IE: Extract from Sln-Engine.lua 
 projectDependencies = { "VisionDLL" .. buildSystemSuffix }
 dofile = "../shared/projects/VisionEnginePlugin/Prj-VisionEnginePlugin.lua"


  • Related flags: NoLinkLibDependencies
